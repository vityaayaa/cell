# CELL — Приложение для сотрудников строительного магазина

Помогает работать со стеллажом, управлять остатками и заявками на склад.

---

## Суть приложения

Строительный магазин. Стеллаж из N×N ячеек, каждая — под конкретный товар (доски, плинтус, вагонка, трубы, наличники). Нужно периодически пополнять со склада.

**Старый процесс:** сотрудник с бумагой ходит вдоль стеллажа, на глаз оценивает → едет на склад → грузит → везёт → раскладывает.

**Проблемы:**
1. Не считалась вместимость ячейки — непонятно, сколько ТОЧНО нужно
2. Бумажные записи теряются, путаются
3. Неудобно отмечать погружённые товары
4. Нет порядка укладки (длинные позиции — вниз для устойчивости)
5. Нет истории

---

## Роли

### Админ
- Настройка стеллажа (N столбцов × N строк)
- Разделение / объединение ячеек
- Назначение товаров в ячейки
- Настройка каталога товаров
- Просмотр истории

### Сотрудник
- Внесение остатков
- Создание и корректировка заявки на склад
- Работа с чеклистом (отмечать взятое)

---

## Логика работы (последовательность)

1. **Внесение остатков** — тап на ячейку → ввод:
   - штучный товар (доски, трубы): число
   - навалом (сыпучие, мелкие): слайдер с делениями (= пачки/ёмкости)

2. **Генерация заявки** — дефицит = вместимость − остаток → пересчёт в пачки

3. **Корректировка черновика** — убрать позицию, изменить объём

4. **Финализация заявки** — фиксация, история

5. **Чеклист** — список позиций со статусами, фильтрация (по длине и т.д.)

---

## Расчёт вместимости

Каждый товар: ширина × высота × длина (мм). Каждая ячейка: ширина × высота × глубина (мм).

Приложение считает:
- сколько влезет по ширине
- сколько влезет по высоте
- итого = строк × столбцов

**Поворот:** если 50×100 мм не влезает — проверяет 100×50 мм. Если влезает — добавляет к вместимости.

**Ручное переопределение:** менеджер вводит число вручную — оно перекрывает расчёт.

---

## Онлайн и офлайн

Приложение **online-first** (S09): интернет есть почти всегда, данные сразу синхронизируются с облаком (Supabase — источник правды). Офлайн — страховка: при потере связи всё пишется локально и досинхронизируется при восстановлении. Несколько пользователей (S07) работают одновременно, каждый со своего телефона, в разных сессиях — один обход в системе как бизнес-правило, поэтому конфликтов нет. Браузерное приложение, ставится как PWA.

# Законы работы Claude в этом проекте

## Глобальные

### 1. Антигаллюцинация
- Утверждаешь факт о библиотеке, API, цифре, ограничении — ОБЯЗАН проверить через Context7 или Exa. Не полагайся на обучающие данные для версий, лимитов, синтаксиса.
- Если не уверен — говори "не уверен, проверю" и проверяй. Никогда не говори уверенно то, в чём есть сомнение.
- Запрещено выдумывать названия npm-пакетов, функций API, параметров конфигов.

### 2. Самокритика
- После каждого предложения/решения — задай себе вопрос "А что если я ошибаюсь?" и озвучь главное возражение против своего же ответа.
- Если решение кажется очевидным — это сигнал проверить его сильнее. Простые решения часто скрывают скрытые компромиссы.
- Явно указывай trade-offs для каждого выбора. Нет выбора без цены.

### 3. Поиск перед советом
- Перед рекомендацией технологии/подхода — найти актуальную информацию (Context7 для доков, Exa для сравнений и community feedback, GitHub MCP).
- Искать не только "почему хорошо", но и "почему плохо / когда не стоит использовать".

### 4. Явные предположения
- Если предположение неверное — это меняет весь совет. Лучше уточнить, чем угадать.

### 5. Конкретность вместо абстракции
- Запрещены фразы: "можно рассмотреть", "зависит от ситуации", "есть несколько подходов" без последующего конкретного выбора с обоснованием.
- Всегда давать рекомендацию, а не меню. Меню — когда реально равнозначные варианты.

### 6. Развитие через вопросы
- В конце каждого блока задавать 1–2 уточняющих вопроса, которые могут изменить направление. Не риторических — реально важных для следующего шага.
- Если пользователь не знает ответа — это информация: значит, нужно обсудить именно это.

### 7. Критика ответов пользователя
- Если ответ пользователя кажется неверным, противоречивым или опасным для проекта — НЕ соглашаться автоматически. Оспорить прямо.
- Фраза "если не усложнит" от пользователя — сигнал к разбору: усложнит или нет. Не принимать как согласие реализовывать.
- Задавать уточняющие вопросы, влияющие на спецификацию, проактивно — не ждать, пока пользователь сам попросит.

## Переопределение правила активации скиллов

**Правило "1% вероятности" из superpowers — ОТМЕНЕНО для этого проекта.**

Скиллы активируются только при явной необходимости (>15% уверенность в пользе) или по явному запросу пользователя. Автоматическая активация при малейшем намёке — лишняя трата контекста. Инструкции пользователя в CLAUDE.md имеют приоритет над поведением скиллов.

### ОБЯЗАТЕЛЬНЫЕ скиллы в конце каждой сессии реализации

**НЕЛЬЗЯ ПРОПУСКАТЬ. БЕЗ ИСКЛЮЧЕНИЙ.**

После git push в каждой сессии I-0N вызвать ОБА скилла по порядку:

1. `/superpowers:finishing-a-development-branch`
2. `/superpowers:verification-before-completion`

Это прописано в каждом промпте в разделе "Скиллы для этой сессии". Пропуск = незавершённая сессия.

## Мета-правила

### Обязательное чтение файлов перед сессией

В каждом промпте сессии есть раздел `## Прочитай перед началом` со списком файлов.  
**ОБЯЗАТЕЛЬНО прочитать ВСЕ файлы из этого списка до начала разговора с пользователем.** Без исключений. Нет частичного чтения. Если список содержит 4 файла — читать все 4.

Нарушение = неполный контекст = неверные ответы и потраченное время пользователя.

### Инструменты (когда использовать)
- **Context7**: любой вопрос про библиотеку, версию, синтаксис API
- **Exa Search**: сравнение подходов, community опыт, "почему X плохо", аналоги
- **GitHub MCP**: поиск примеров реализаций, репозиториев
- **Supabase MCP**: только когда схема зафиксирована и нужна реализация
- **Vercel MCP**: только на этапе деплоя

### Когда остановиться и спросить
- Если два варианта реально равнозначны — не выбирать самому, описать trade-off и спросить.
- Если обнаружено противоречие между желаемым и технически разумным — явно назвать противоречие, не замалчивать.
- Если сессия уходит в другую тему — остановить и спросить: "Переходим или возвращаемся?"

### Вопросы пользователю
- Задавать несколько вопросов сразу, не по одному — кроме случаев, когда следующий вопрос зависит от ответа предыдущего.
- Нет смысла тратить сообщения на последовательные вопросы, если они независимы.

### Честность
- Если не знаешь — говори "не знаю, проверю" или "не знаю, вот где найти".
- Если предыдущий ответ был неверен — исправить явно: "Я ошибся в предыдущем ответе. Верно вот что."
- Не защищать свои прошлые ответы из-за инерции.

### Пользователь — не кодер
Пользователь не пишет код вообще. Псевдокод, формулы вида `floor(a / b)`, синтаксис переменных — ничего не значат и не помогают. Объяснять только словами и примерами с реальными числами. Если словами не объяснить — не объяснять вовсе.

### Проверка следующего промпта перед завершением сессии
После git push — ВСЕГДА читать промпт следующей сессии. Найти вопросы, уже решённые в текущей или прошлых сессиях, и добавить их в секцию "Уже закрыто". Не ждать, пока пользователь попросит об этом — делать автоматически каждый раз.

### Приносить больше, чем написано в промпте
Блок "принеси в разговор" — минимум, не потолок. После исследования задать себе вопрос: "что здесь важного и неочевидного, чего нет в промпте?" Быть инициативным партнёром, а не исполнителем инструкций.

### Один блок — один вопрос "Переходим?"
Не совмещать несколько крупных решений в одном ответе без явного запроса. После каждого завершённого блока спрашивать "Переходим к следующему?" — пользователь хочет итерационное обсуждение, а не монолитные ответы.

### Задавать все вопросы сразу
Перед написанием спеки — пройтись по всем неясностям и задать все вопросы в одном сообщении. Не ждать напоминания. Пользователь не будет повторять это правило каждую сессию.
