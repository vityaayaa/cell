# S07 — Роли и доступ

## Принятые решения

### Два типа ролей

`admin` и `employee`. Достаточно для команды 2–5 человек.

| Роль | Кто |
|------|-----|
| `admin` | Директор, менеджер — полный доступ |
| `employee` | Сотрудник — только операционные задачи |

---

### Аутентификация

Supabase Auth: email + password. Каждый заходит со своего телефона. Саморегистрации нет — только admin создаёт аккаунты.

**Email — только реальный адрес.** Supabase использует email для подтверждения аккаунта и сброса пароля. При создании аккаунта admin-ом сотрудник получает invite-письмо, устанавливает свой пароль и впредь управляет им самостоятельно через "Забыл пароль?" на экране входа. Это обеспечивает автономность: если admin недоступен, сотрудники могут самостоятельно восстановить доступ.

**Автовыход по бездействию — нет.** Ручной выход из аккаунта. На личных телефонах оставаться залогиненным постоянно — норма.

**Первый запуск:** onboarding-экран, где создаётся первый аккаунт admin (Edge Function без авторизации, защищена проверкой "ни одного admin ещё нет"). После этого admin создаёт остальных пользователей из настроек.

---

### Управление аккаунтами

Только admin. Из настроек приложения:

| Действие | Описание |
|----------|---------|
| Создать | Имя, email, роль → Supabase отправляет invite-письмо; сотрудник сам устанавливает пароль |
| Заблокировать | Пользователь не может войти; данные сохраняются |
| Разблокировать | Восстановить доступ |
| Удалить | Аккаунт удаляется; записи в истории, логах — остаются, привязаны к ID |

---

### Параллельная работа

Каждый заходит со своего телефона → несколько пользователей одновременно — нормально.

Ограничение **"один sweep одновременно"** остаётся бизнес-правилом (S04/S05). Если кто-то уже ведёт обход — новый sweep начать нельзя.

**Менеджер сессий при конфликте:**

| Роль | Что видит и может |
|------|-------------------|
| Employee | Кто ведёт обход, когда начал, прогресс. Кнопка [Продолжить] — только если это его обход. Чужой sweep: ждать. |
| Admin | То же + кнопка [Завершить принудительно] — переводит чужую сессию в `abandoned`. |

---

### История сессий

Интегрирована в экран «Сессии» (S14) — единый для обеих ролей. Активные и завершённые сессии отображаются на одном экране.

| Роль | Что видит |
|------|-----------|
| Admin | Все сессии всех пользователей + агрегаты по товарам + экспорт Excel |
| Employee | Все сессии всех пользователей (только просмотр) |

История неизменяема — редактировать нельзя никому.

---

### Подтверждение необратимых действий

Экран подтверждения ("Это нельзя отменить. Продолжить?") для:

| Действие | Кто |
|----------|-----|
| Финализация заявки (S05) | Admin, Employee |
| Принудительный abandon чужой сессии | Admin |
| Удаление товара из каталога (S01) | Admin |
| Удаление / объединение ячейки с товаром (S02) | Admin |
| Удаление / блокировка пользователя | Admin |

---

### Audit log

Только admin. Лог всех значимых действий с фильтрацией.

**Что логируется:**

| Категория | События |
|-----------|---------|
| Аутентификация | Вход, выход |
| Аккаунты | Создан / заблокирован / разблокирован / удалён пользователь |
| Каталог | Товар добавлен / изменён / удалён; Материал добавлен / изменён / удалён |
| Стеллаж | Ячейка разделена / объединена; Товар назначен на ячейку / снят; Добавлена строка / столбец; Вместимость переопределена; Флаг `needs_review` подтверждён |
| Обход | Начат / завершён / заброшен (при принудительном — кто инициировал) |
| Остатки | Введён / исправлен остаток (ячейка, значение, обход) |
| Заявка | Строка удалена / изменена / добавлена в черновике; Заявка финализирована |
| Чеклист | Строка отмечена (статус, `actual_packs`, кто) |

**Фильтры:** по пользователю, по категории события, по дате (диапазон).

## Модель данных

### UserProfile

Supabase Auth хранит email и password. Приложение добавляет профиль:

| Поле | Тип | Описание |
|------|-----|---------|
| `id` | uuid | FK → auth.users |
| `name` | string | Имя для отображения ("Иван", "Менеджер") |
| `role` | enum | `admin` \| `employee` |
| `is_active` | boolean | false = заблокирован |
| `created_at` | timestamp | |
| `created_by` | FK → UserProfile | Кто создал; null для первого admin |

### AuditLog

| Поле | Тип | Описание |
|------|-----|---------|
| `id` | uuid | |
| `actor_id` | FK → UserProfile | Кто совершил действие |
| `event_type` | enum | Тип события (см. таблицу выше) |
| `entity_type` | string | Тип объекта: `product`, `cell`, `session`, `user`, etc. |
| `entity_id` | uuid | ID затронутого объекта |
| `old_value` | jsonb | Предыдущее состояние; null для create-событий |
| `new_value` | jsonb | Новое состояние; null для delete-событий |
| `created_at` | timestamp | |

## Что намеренно за бортом

| Что | Почему |
|-----|--------|
| Несколько магазинов | Преждевременно для v1 |
| Роль "только чтение" | Директор = admin |
| Промежуточные роли | Двух ролей достаточно |
| Автовыход по бездействию | Личные телефоны; нет практической нужды |
| Двухфакторная аутентификация | Нет нужды в малой команде |
| Саморегистрация | Только admin создаёт аккаунты (но сотрудники самостоятельно управляют паролями) |
| SSO / OAuth | Не нужно для внутреннего приложения |
| Фиктивные email (`ivan@cell.local`) | Убраны: нужны реальные email для автономного сброса пароля |
