# S10 — Стек и архитектура

Фундамент построен на решениях S07 (бэкенд Supabase) и S09 (online-first, Dexie, PWA, Realtime, LWW). Здесь фиксируется полный стек, архитектура модулей, структура проекта, тесты и деплой.

└───────────────▲────────────────────────────┬───────────────┘
                │                            │
┌───────────────┴────────────────────────────▼───────────────┐

└─────────────────────────────────────────────────────────────┘

  │  Доменное ядро (чистый TS, без сети и UI):                │

  │  → вызывается из UI и слоя синхронизации, плотно тестится │
  └──────────────────────────────────────────────────────────┘

### Как текут данные

> **Уточнение к S09.** В S09 чтение описано как «с сервера». Это про источник правды (Supabase), а не про механизм. Механизм такой: экран читает из локальной витрины, которую слой синхронизации непрерывно держит свежей из Supabase. Источник правды — по-прежнему сервер; витрина — лишь всегда-актуальная поверхность чтения. Противоречия нет.

- **Чтение.** Экран всегда читает из витрины (Dexie через `useLiveQuery`) — мгновенно, работает и офлайн.
- **Свежесть.** При старте слой синхронизации грузит всё из Supabase в витрину; держит Realtime-подписки и обновляет витрину при изменениях. Онлайн при заходе всегда подтягивается актуальное из базы.
- **Запись (онлайн).** Пишем в Supabase → при успехе обновляем витрину → UI обновляется сам (через liveQuery).
- **Запись (офлайн).** Кладём операцию в очередь (Dexie) + оптимистично обновляем витрину. При возврате связи очередь уходит на Supabase по порядку, идемпотентно (upsert по ID записи).
- **Конфликты.** Last-Write-Wins по `updated_at` (S09).
- **Эхо.** Своё же изменение, вернувшееся через Realtime, отбрасываем сравнением ID/`updated_at` — без двойного применения.
- **Вычисления.** Доменное ядро — чистые функции; одни и те же и при вводе остатков, и при настройке ячеек.

### Модули

1. **`domain/`** — чистая логика (вместимость, BSP, заявка). Не зависит от React и Supabase. Главный объект тестирования.
2. **`data/`** — схема Dexie, клиент Supabase, слой синхронизации (очередь, Realtime, определение связи).
3. **`features/`** — экраны по доменам (каталог, стеллаж, остатки, заявка, чеклист, история, аккаунты, аудит).
4. **`ui/`** — переиспользуемые компоненты (shadcn/ui + свои) и анимации.
5. **`app/`** — маршрутизация, провайдеры, ролевые оболочки, onboarding первого админа.

## Стратегия тестирования

| Уровень | Что | Инструмент | Обязательность |
|---------|-----|-----------|:---:|
| Доменное ядро | Формула вместимости с поворотом (S03), BSP-пересчёт (S02), расчёт дефицита/пачек/пограничных (S01/S05) | Vitest | **Обязательно** |
| Компоненты | Нетривиальные экраны (ввод остатков, черновик заявки) | Vitest + Testing Library | По мере надобности, точечно |
| Сквозные (e2e) | Полный сценарий в браузере | — | **Отклонено в v1** |

Сквозную проверку всего сценария владелец проходит вручную перед релизом. Юнит-тесты гоняются в CI на каждый push; красные тесты блокируют деплой.

## CI/CD

- **Репозиторий:** один, GitHub (приватный).
- **На каждый push:** GitHub Actions запускает `typecheck` + `test`.
- **Vercel:** подключён к репозиторию. Push в `main` → продакшн-деплой; ветки/PR → preview-деплой (открыть и проверить до слияния).
- **Supabase:** миграции из `supabase/migrations` применяются к проекту (сначала через CLI, позже можно автоматизировать).
- **Секреты:** URL и публичный ключ Supabase — в переменных Vercel. Служебный ключ — только в Edge Function, никогда в клиенте.

## Ключевые архитектурные решения

| Решение | Суть | Почему |
|---------|------|--------|
| Чтение из витрины | Экран читает из Dexie (liveQuery); сервер+Realtime держат её свежей | Мгновенный UI, работает офлайн; онлайн всегда актуально из БД |
| Вычисления на клиенте | Доменное ядро чистым TS | Работает офлайн и мгновенно; легко тестировать |
| ID создаёт клиент | UUID на телефоне до отправки | Идемпотентная офлайн-очередь без дублей |
| `updated_at` везде | На всех редактируемых таблицах | Last-Write-Wins (S09) |
| Защита прав — 2 слоя | Разный набор экранов по роли + RLS в базе | Удобство (UI) и надёжность (база не пускает в обход) |
| Аудит-лог в базе | Пишут триггеры; приложение показывает «Историю действий» человеческим языком (admin) | Нельзя забыть или подделать; читаемо для владельца |
| Создание аккаунтов | Через Edge Function | Служебный ключ не попадает в клиент |
| Удаление товара | Снимается со всех ячеек, им ставится флаг «проверь» (S03) | Не оставлять ячейки со ссылкой в пустоту |

## Что намеренно отклонено

| Что | Почему |
|-----|--------|
| Svelte / Vue | React — за экосистему и надёжность ИИ-генерации; вес для установленного PWA некритичен |
| Next.js / SSR | Приложение за логином, SEO не нужно; SPA проще |
| Монорепо / Turborepo | Одно приложение; усложнение без пользы (YAGNI) |
| TanStack Query | Не нужен: чтение идёт из Dexie через liveQuery, компоненты не ходят на сервер напрямую |
| RxDB / PouchDB | Отклонены в S09 — Dexie проще, без проприетарного бэкенда |
| Playwright e2e | Ручная сквозная проверка в v1 |
| Redux / тяжёлый state | Server-данные живут в Dexie; для UI-состояния хватает Zustand |
| Background Sync, CRDT, SQLite/OPFS | Отклонены в S09 |
