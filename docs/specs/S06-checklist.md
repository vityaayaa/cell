# S06 — Чеклист

## Источник чеклиста

Зафиксировано в S05. Чеклист создаётся автоматически при финализации заявки (переход `ordering` → `fulfilling`). Каждая строка чеклиста = одна строка заявки (`OrderLine`). Заявка после этого locked.

---

## Принятые решения

### Статусы строк

Зафиксировано в S05: `pending` | `done` | `unavailable`.

| Статус | Значение |
|--------|---------|
| `pending` | Не обработано |
| `done` | Товар взят (полностью или частично) |
| `unavailable` | Товара нет на складе |

Любой статус обратим до перехода сессии в `completed`. После — история зафиксирована и не редактируется.

---

### Фактическое количество

Поле `actual_packs` фиксирует, сколько пачек реально взяли.

- При `pending` и `unavailable` — пустое
- При `done` — число от 1 до `quantity_packs` (из `OrderLine`)

Если `actual_packs < quantity_packs` — строка в истории отображается как "заказали N, взяли M". Если равно — просто "взяли N".

**Зачем нужно:** "взял всё" и "взял половину" — разные исходы. Статус `done` без количества не даёт этого различия — менеджер не знает, 10 пачек приехало или 8.

---

### Взаимодействие при отметке

Оптимизировано под работу в перчатках. Стандартный случай — один тап, исключение — один дополнительный шаг.

| Действие | Шаги |
|----------|------|
| Взял всё | Один тап → `done`, `actual_packs = quantity_packs` |
| Взял меньше | Тап → открывается поле с предзаполненным числом → "−" / "+" или прямой ввод → подтвердить |
| Нет на складе | Отдельное действие → `unavailable`, `actual_packs` не заполняется |

Прямой ввод числа обязателен — для случаев когда разница большая и жать "−" 30 раз неудобно.

Точная реализация кнопок и жестов — в S13.

---

### Порядок строк

Базовая сортировка — из S05: материал (выбранный первым блоком, остальные по алфавиту) → длина убывает → название А→Я.

Дополнительно: строки со статусом `done` и `unavailable` автоматически смещаются в конец списка. Внутри "погруженных" сортировка сохраняется.

Отдельного фильтра нет — чеклист 10–20 позиций, сортировка уже группирует по материалу и длине.

---

### Счётчик прогресса

Вверху чеклиста всегда показывается: "Взято N из M".

- M = все строки чеклиста
- N = строки в статусе `done` или `unavailable`

---

### Завершение сессии

Сессия переходит в `completed` автоматически, когда все строки чеклиста в статусе `done` или `unavailable`. Зафиксировано в S05.

---

### Повторный визит

Сессия в статусе `fulfilling` висит открытой без таймаута. Сотрудник уехал со склада, вернулся на следующий день — открывает ту же сессию, продолжает с того места, где остановился.

Новый обход (`sweeping`) возможен параллельно — сессии в `fulfilling` его не блокируют (S05).

---

### Офлайн

Каждое действие в чеклисте (смена статуса, ввод `actual_packs`) записывается локально немедленно. При появлении связи — очередь операций уходит на сервер в порядке создания. Каждая операция идемпотентна.

CRDT не нужен: один сотрудник работает с одним чеклистом, конкурентных конфликтов нет. Если в будущем двое сотрудников получат разные заявки — у каждого свой чеклист, пересечений нет.

---

### Бумажный экспорт

Из чеклиста доступна печать PDF-бланка:

- Та же сортировка, что в чеклисте на момент экспорта (погруженные в конце)
- Для каждой строки: название товара, количество пачек, поля для ручной отметки
- Поля в бланке: галочка "взял", крестик "нет на складе", строка "взял столько: ___"

---

## Модель данных

### Дополнения к ChecklistEntry (S05)

Добавляется одно поле:

| Поле | Тип | По умолч. | Описание |
|------|-----|:---------:|---------|
| `actual_packs` | integer | null | Фактически взятых пачек; null если `pending` или `unavailable` |

**Полная модель ChecklistEntry:**

| Поле | Тип | Описание |
|------|-----|---------|
| `id` | uuid | |
| `order_line_id` | FK → OrderLine | |
| `status` | enum | `pending` \| `done` \| `unavailable` |
| `actual_packs` | integer | null пока не обработано или нет на складе |
| `updated_at` | timestamp | |
| `user_id` | FK → User | Кто отметил |

**Инварианты:**
- `status = done` → `actual_packs` ∈ [1, `order_line.quantity_packs`]
- `status = unavailable` → `actual_packs IS NULL`
- `status = pending` → `actual_packs IS NULL`

---

## Что намеренно за бортом

| Что | Почему |
|-----|--------|
| Фильтр по материалу / длине | 10–20 позиций в чеклисте — фильтр избыточен; сортировка уже группирует |
| Замена позиции на аналог | Нет справочника аналогов; выходит за рамки v1 |
| Адрес хранения на складе | Адресация хаотичная, сотрудник знает расположение сам |
| CRDT для синхронизации | Один сотрудник = нет конкурентных конфликтов |
| Таймаут сессии `fulfilling` | Нет практической нужды; повторный визит — штатный сценарий |
