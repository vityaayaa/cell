# S09 — Офлайн и синхронизация

## Принятые решения

### Архитектура: online-first с офлайн-резилиентностью

Приложение работает в режиме **online-first**: Supabase — источник правды, данные всегда подтягиваются с сервера. Офлайн — редкий крайний случай, а не основной режим работы.

Это не офлайн-first. Разница принципиальная:

| | Online-first (наш выбор) | Offline-first |
|---|---|---|
| Источник правды | Supabase | IndexedDB |
| Чтение | С сервера (кэш при офлайн) | Всегда из IndexedDB |
| Сложность | Низкая | Высокая |
| Подходит когда | Интернет почти всегда есть | Интернет редко есть |

---

### PWA

Приложение устанавливается на экран телефона как PWA (Progressive Web App). Реализация через **vite-plugin-pwa**.

Что это даёт:
- Иконка на экране, запускается как обычное приложение
- Статика кэшируется Service Worker — приложение открывается мгновенно
- Работает на Android и iOS (Safari 11.1+)

Первый запуск всегда требует интернета — для авторизации и загрузки данных. Это приемлемо.

---

### Service Worker

Режим: `generateSW` (автоматическая генерация, vite-plugin-pwa).

| Тип ресурса | Стратегия | Почему |
|-------------|-----------|--------|
| JS, CSS, шрифты, иконки | CacheFirst | Не меняются между сборками — нужна скорость |
| index.html (app shell) | NetworkFirst | Нужна актуальная версия; кэш — если офлайн |
| Запросы к Supabase API | NetworkOnly | Данные не кэшируются SW; за кэш отвечает Dexie.js |

При обновлении приложения новый Service Worker активируется сразу (skipWaiting + clientsClaim).

---

### Локальное хранилище: Dexie.js

Библиотека: **Dexie.js** (IndexedDB wrapper). Используется как локальный кэш и буфер офлайн-операций.

Dexie.js выбран потому что:
- Простой API, минимум бойлерплейта
- Транзакции IndexedDB — атомарность при записи
- Не требует проприетарного бэкенда (в отличие от RxDB, PouchDB)
- Малый размер, хорошая поддержка браузеров

IndexedDB — кэш, не источник правды. При конфликте между локальными данными и Supabase побеждает Supabase.

**Хранимые данные в IndexedDB:**

| Таблица | Что хранится |
|---------|-------------|
| `products` | Полный каталог товаров |
| `materials` | Справочник материалов |
| `cells` | Все ячейки стеллажа |
| `sessions` | Активные и недавние сессии |
| `stock_entries` | Остатки текущего обхода |
| `orders` | Заявки |
| `order_lines` | Строки заявок |
| `checklist_entries` | Записи чеклиста |
| `sync_queue` | Очередь офлайн-операций |

### Стратегия синхронизации

#### При открытии приложения

1. Показывается последний кэш из IndexedDB — приложение реагирует мгновенно
2. Параллельно: запрос к Supabase, полная загрузка всех данных
3. IndexedDB обновляется (bulkPut) — UI перерисовывается если что-то изменилось
4. Запускаются Realtime-подписки

Объём данных маленький (< 1 МБ), полная загрузка при каждом открытии — нормально.

#### Пока приложение открыто: Supabase Realtime

Supabase Realtime (WebSocket) — изменения с других устройств прилетают моментально без перезагрузки.

Подписки на таблицы:
- `products`, `materials` — если admin меняет каталог
- `cells` — если admin меняет стеллаж
- `sessions` — смена статуса активной сессии
- `checklist_entries` — если двое работают с разными чеклистами одновременно

При получении события от Realtime: обновляется IndexedDB и UI.

#### При каждой записи (онлайн)

1. IndexedDB обновляется оптимистично — немедленно
2. Запрос уходит на Supabase
3. При успехе — ничего дополнительного (IndexedDB уже актуален)
4. При ошибке — IndexedDB откатывается к предыдущему состоянию + Toast: "Не сохранилось. Попробуйте ещё раз."

---

### Офлайн-очередь

При потере связи запись идёт в `sync_queue` в IndexedDB.

Структура записи в очереди:

| Поле | Тип | Описание |
|------|-----|---------|
| `id` | uuid | |
| `table` | string | Таблица: `stock_entries`, `checklist_entries`, etc. |
| `operation` | enum | `insert` \| `update` \| `delete` |
| `payload` | json | Данные для отправки |
| `created_at` | timestamp | Порядок выполнения |
| `retry_count` | integer | Количество попыток |

При восстановлении связи:
1. Очередь обрабатывается по порядку (по `created_at`)
2. Каждая операция идемпотентна (upsert по id)
3. После отправки всей очереди — полная синхронизация с Supabase

Background Sync API (синхронизация в фоне, когда вкладка закрыта) не используется — не поддерживается на Safari/iOS и Firefox. Синхронизация происходит только при открытом приложении.

---

### Определение связи

`navigator.onLine` ненадёжен: браузер может сообщать "online", когда реального доступа к интернету нет. Поэтому:

1. `navigator.onLine = false` → офлайн (надёжный сигнал об отсутствии связи)
2. `navigator.onLine = true` → **не гарантирует** связь; проверяется реальным ping к Supabase
3. Ping = лёгкий запрос к Supabase health endpoint при каждой попытке записи

Состояния в приложении: `online` | `offline` | `syncing`. Отображается в UI.

Пока в `sync_queue` есть необработанные записи — офлайн-индикатор показывает "N записей ожидают синхронизации" (N = количество записей в очереди). Это мотивирует пользователя открыть приложение при появлении связи.

---

### Конфликты

Стратегия: **Last-Write-Wins** по полю `updated_at`.

При конфликте между локальным и серверным значением побеждает то, у которого `updated_at` новее. Для нашего сценария этого достаточно:

- Обход (sweep) — одновременно может быть только один → конфликтов нет
- Чеклист — двое работают с разными чеклистами → пересечений нет
- Каталог/стеллаж — только admin, последовательные правки → конфликты крайне редки
- При редком конфликте (admin меняет стеллаж пока employee вводит остаток): побеждает тот, кто сохранил позже

CRDT не нужен.

**Требование к модели данных:** поле `updated_at` (timestamp) обязательно на всех таблицах, которые могут редактироваться. Детали — в S10 (схема БД).

---

### Первый запуск и смена устройства

**Первый запуск:** требует интернета. Пользователь авторизуется, данные загружаются с Supabase, IndexedDB заполняется.

**Смена устройства или потеря телефона:** пользователь входит по email/паролю (S07), данные загружаются с Supabase. Локальный кэш строится заново автоматически.

**Safari ITP (7-дневное удаление):** если пользователь не открывал приложение 7+ дней, Safari может удалить IndexedDB. При следующем открытии (с интернетом) данные восстанавливаются с Supabase автоматически — для пользователя незаметно.

Для защиты от удаления вызывается `navigator.storage.persist()` при установке PWA. На Chrome это гарантирует постоянное хранилище. На Safari — не гарантирует, но попытка делается.

## Поведение по сценариям

| Сценарий | Что происходит |
|----------|---------------|
| Открыл с интернетом | Кэш → мгновенный UI, потом полная синхронизация с Supabase |
| Открыл без интернета | Показывает последний кэш, баннер "нет связи" |
| Потерял связь в процессе | Действия идут в offline-очередь; UI показывает "офлайн" |
| Связь вернулась | Очередь уходит на Supabase, данные синхронизируются |
| Admin изменил стеллаж на своём телефоне | Realtime-событие прилетает к employee за < 1 сек |
| Двое работают с разными чеклистами | Каждый видит свой; изменения другого прилетают через Realtime |
| Новый телефон / потеря данных Safari | Вход → полная загрузка с Supabase |

---

## Что намеренно за бортом

| Что | Почему |
|-----|--------|
| Офлайн-first архитектура | Интернет почти всегда есть; online-first проще |
| Background Sync API | Не поддерживается на Safari/iOS и Firefox |
| CRDT | Один сотрудник на одну операцию; LWW достаточен |
| SQLite via OPFS | Избыточно; IndexedDB справляется с нашим объёмом данных |
| Инкрементальная синхронизация | Объём < 1 МБ; полная загрузка при открытии — нормально |
| Кэширование API-запросов в Service Worker | За кэш отвечает Dexie.js, не SW |
| Выбор разрешения конфликта пользователем | LWW достаточен; диалог конфликтов избыточен |
