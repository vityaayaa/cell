# S05 — Заявка на склад

## Концепция: Сессия

Вся работа организована вокруг **сессии** — единицы от первого шага обхода до последней отметки в чеклисте.

Сессия = Обход + Заявка + Чеклист. Одна запись в истории = одна завершённая сессия.

---

## Жизненный цикл сессии

```
sweeping ↔ ordering → fulfilling → completed
    │           │           │
    └───────────┴───────────┴──→ abandoned
```

| Статус | Что происходит |
|--------|---------------|
| `sweeping` | Идёт обход, вводятся остатки (соответствует S04: `in_progress` / `review`) |
| `ordering` | Черновик заявки открыт для редактирования |
| `fulfilling` | Заявка финализирована, чеклист активен |
| `completed` | Все позиции чеклиста отмечены → сессия уходит в историю |
| `abandoned` | Сессия брошена на любом этапе → уходит в историю как есть |

**Переходы:**
- `sweeping` → `ordering`: нажата кнопка "Перейти к заявке"
- `ordering` → `sweeping`: нажата кнопка "Вернуться к обходу"
- `ordering` → `fulfilling`: финализация заявки (одностороннее, назад нельзя)
- `fulfilling` → `completed`: все позиции отмечены (выполнено или нет на складе)
- любой → `abandoned`: начата новая сессия поверх текущей

**Параллельные сессии:**
- Одновременно может быть только одна сессия в фазе `sweeping`
- Сессии в фазе `fulfilling` могут висеть открытыми сколько угодно — они не блокируют новый обход
- Пример: сессия 1 в `fulfilling` (складовщик работает) + сессия 2 в `sweeping` (менеджер идёт по стеллажу) — нормально

---

## Генерация заявки

При переходе из `sweeping` в `ordering` система автоматически формирует черновик по всем ячейкам текущего обхода.

**Формула по типу товара:**

Для `unit` и `round`:
```
deficit_units = capacity_units - current_stock_units
full_packs    = floor(deficit_units / pack_size)
```

Для `bulk` (capacity и stock уже в пачках — деление на pack_size не применяется):
```
deficit_packs = capacity_packs - current_stock_packs
full_packs    = deficit_packs
```

**Почему агрегация, а не поячеечный расчёт:** суммирование `floor(deficit_i / pack_size)` по ячейкам теряет дробные остатки. Пример: дефицит 7 и 6 шт при pack_size=10 — поячеечно даёт 0+0=0 пачек; агрегация даёт `floor(13/10)=1` пачку (верно). Агрегация всегда ≥ поячеечного результата — никогда не занижает заявку.

**Правила включения в заявку:**

| Условие | Что происходит |
|---------|---------------|
| `full_packs ≥ 1` | Позиция попадает в основную часть заявки |
| `full_packs = 0`, дефицит есть | Пограничная позиция — в отдельный блок внизу |
| `deficit_units = 0` | Позиция не включается |

**Агрегация:** Один товар = одна строка в заявке, независимо от того, в скольких ячейках он назначен. Суммируются дефициты по всем ячейкам с этим товаром, затем применяется формула.

Ячейки без назначенного товара — игнорируются. Ячейки, не затронутые в текущем обходе — не включаются в заявку.

## Черновик заявки (фаза `ordering`)

### Что можно делать

| Действие | Описание |
|----------|---------|
| Убрать строку | Позиция исключается из заявки |
| Изменить количество пачек | Ввести другое число вручную (вместо расчётного) |
| Активировать пограничную позицию | Переключатель → позиция перемещается из нижнего блока в основную заявку |
| Добавить строку вручную | Выбрать товар из каталога и указать количество пачек (для разовых позиций не со стеллажа) |

### Что нельзя
- Добавить товар, которого нет в каталоге (только из каталога)
- Изменить формулу расчёта дефицита

### Содержимое строки заявки

| Поле | Описание |
|------|---------|
| `product_id` | Товар из каталога |
| `quantity_packs` | Количество пачек (расчётное или введённое вручную) |
| `quantity_units` | Штук = `quantity_packs × pack_size` (для справки) |
| `deficit_units` | Исходный расчётный дефицит в штуках (только для автоматических строк) |
| `is_manual` | true = добавлено вручную |
| `is_boundary` | true = активированная пограничная позиция |

---

## Финализация

Переход `ordering` → `fulfilling`. Необратимо.

После финализации:
- Заявка фиксируется (locked), изменения невозможны
- Создаётся чеклист из строк заявки
- Сессия переходит в статус `fulfilling`

---

## Чеклист (фаза `fulfilling`)

Подробный UI — в S13. Здесь фиксируются принципиальные решения.

Каждая строка чеклиста = строка заявки. Статусы строки:

| Статус | Значение |
|--------|---------|
| `pending` | Не обработано |
| `done` | Товар взят |
| `unavailable` | Нет на складе |

Сессия переходит в `completed` когда все строки в статусе `done` или `unavailable`.

Данные о выполнении (какие строки и с каким статусом) сохраняются в истории вместе с сессией.

## Сортировка (для заявки и чеклиста)

Одна и та же сортировка, одни и те же настройки.

**По умолчанию:** Дерево → длина убывает → название А→Я

**Логика:**
1. **Материал** — выбранный материал первым блоком, остальные по алфавиту
2. **Внутри блока — длина товара** (от длинных к коротким)
3. **Внутри одной длины — полное отображаемое имя** (А → Я: "Брусок 20×30×3000" < "Брусок 50×50×3000" < "Доска 20×100×3000")

Выбор материала-приоритета — настраивается в интерфейсе (по умолчанию: Дерево).

**Дефицит — не сортировка, а визуальный маркер.** Строка с нулевым остатком выделяется цветом независимо от позиции в списке.

## Что намеренно за бортом

| Что | Почему |
|-----|--------|
| Одновременный sweep двух сотрудников | Офлайн-первое приложение, один пользователь — реалтайм-синхронизация не в v1 |
| Статус "товар уже едет" | Операционная дисциплина: новый обход не начинается пока не разложили предыдущий завоз |
| Расфинализация заявки | Необратимо по дизайну |
| Агрегация дефицита по поставщику | Нет поставщиков в модели данных v1 |
