# S04 — Внесение остатков

## Принятые решения

### Обход (sweep) — единица работы

Ввод остатков происходит только в рамках **обхода**. Spot-обновлений (одна ячейка без обхода) нет.

**Поток:**
```
Начать обход → вводить остатки → Перейти к заявке → Финализировать заявку
                    ↑                     ↕
                    └──────────────────────┘
                    (можно возвращаться)
```

Между обходом, черновиком заявки и чеклистом — свободная навигация в обе стороны. Обход закрывается не при "завершить", а при финализации заявки.

---

### Один обход одновременно

В системе может быть только один активный обход. При попытке начать новый — показывается **менеджер сессий**:

```
Незавершённый обход
Начат 2 июня, 3 дня назад
Обновлено 12 из 45 ячеек

[Продолжить]   [Начать новый]
```

"Начать новый" → старый обход переходит в `abandoned`. Данные об остатках из него **не откатываются** — всё, что было посчитано, остаётся в базе.

### Явная отмена сессии

Сотрудник может отменить любую активную сессию прямо из менеджера сессий (S14), без необходимости начинать новую. Кнопка "Отменить" — деструктивная, на карточке сессии.

Подтверждение: "Сессия будет отменена. Введённые данные обхода сохранятся."

Сессия переходит в `abandoned`. Данные не откатываются. Доступно для всех фаз (`sweeping`, `ordering`, `fulfilling`).

Таймаута нет. Обход остаётся `in_progress` до явного действия пользователя.

---

### Состояния обхода

Состояния управляются через `Session.status` (полная машина состояний — в S05). Для фазы обхода актуальны два статуса:

| Session.status | Что происходит |
|----------------|---------------|
| `sweeping` | Идёт ввод остатков |
| `ordering` | Черновик заявки открыт; можно вернуться к вводу остатков |

Переходы `sweeping ↔ ordering → fulfilling → completed / abandoned` — в S05.

---

### Прогресс обхода

Счётчик "Обновлено N из M ячеек":
- M = количество листовых ячеек с назначенным товаром
- Ячейки без товара — серые, недоступные, в счётчик не входят
- Ячейки, не тронутые в текущем обходе, показывают дату последнего обновления: "последний раз 3 дня назад"

---

### Слепой пересчёт

При открытии ячейки предыдущий остаток **не показывается**. Сотрудник видит пустое поле ввода. Это предотвращает подтверждение старых данных вместо реального подсчёта.

---

### Ввод остатков по типу товара

| Тип | Элемент ввода | Хранимое значение |
|-----|:---:|:---:|
| `unit` (штучный) | Числовое поле (штук) | integer (штук) |
| `round` (круглый) | Числовое поле (штук) | integer (штук) |
| `bulk` (навалом) | Слайдер (% заполненности) | integer (эквивалент пачек) |

**Bulk — детали:** слайдер показывает % наполненности. Вместимость `bulk`-ячейки задаётся в пачках (S03). Хранится эквивалент пачек: `round(fill_percent / 100 × capacity_packs)`.

---

### Валидация при вводе

**Остаток > вместимость** — блокировка. Поле подсвечивается ошибкой, кнопка "Сохранить" неактивна пока число не исправлено.

Повторный ввод в ту же ячейку в рамках одного обхода — разрешён (перезаписывает предыдущее значение).

---

### Данные сохраняются немедленно

Каждый тап на ячейку → ввод → сохранение записывает данные в базу сразу. Нет "корзины" обхода, которую нужно сбрасывать при закрытии. Если приложение закрылось или разрядилось — все введённые остатки сохранены, сессия остаётся в статусе `sweeping`.

---

### Audit trail

Каждая запись об остатке фиксирует: ячейка, значение, дата/время, ID сессии, ID пользователя.

`user_id` включён в модель с первой версии, даже если в v1 работает один пользователь. Это исключает болезненную миграцию при добавлении ролей в S07.

---

## Модель данных

Таблица Sweep упразднена. Группировка записей об остатках осуществляется через `session_id` — ссылку на `Session` (S05).

### StockEntry

| Поле | Тип | Описание |
|------|-----|---------|
| `id` | uuid | |
| `cell_id` | FK → Cell | Листовая ячейка |
| `session_id` | FK → Session | Сессия, в рамках которой сделана запись |
| `user_id` | FK → User | Кто внёс |
| `value` | integer | Остаток (штук или эквивалент пачек для bulk) |
| `created_at` | timestamp | |

Для одной ячейки в одной сессии может быть несколько записей (если ввод корректировался). Актуальная — последняя по `created_at`.

---

## Что намеренно за бортом

| Что | Почему |
|-----|--------|
| Spot-обновления (одна ячейка без обхода) | Нет реального сценария — заявки делаются при заметном дефиците по стеллажу |
| Показ предыдущего остатка при вводе | Слепой пересчёт — сознательное решение |
| Таймаут обхода | Нет практической нужды; дата видна в менеджере сессий |
| Фотофиксация ячейки | Не нужно для v1 |
| Блокировка ввода при `needs_review` флаге | Зафиксировано в S03 — флаг не блокирует |
