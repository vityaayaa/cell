# S02 — Модель стеллажа

## Принятые решения

### Один стеллаж — v1
В системе один стеллаж. Поддержка нескольких стеллажей не реализуется в v1 — это преждевременное усложнение для одного магазина.

### Базовая сетка: равномерная таблица N×M
Стеллаж — это сетка из N строк × M столбцов. Визуально все базовые ячейки равные (горизонтальный/landscape формат, не квадратные). Физические размеры (мм) у каждой ячейки свои и вводятся вручную администратором рулеткой.

Строки: A, B, C... (сверху вниз).  
Столбцы: 1, 2, 3... (слева направо).  
Адрес базовой ячейки: `A1`, `B3`, `C2`.

### Деление ячеек: 2D BSP-дерево
Любую листовую ячейку можно разделить по вертикали (V — левая/правая) или по горизонтали (H — верхняя/нижняя). Каждое деление бинарное: создаёт ровно двух потомков с равными размерами.

**Равномерное распределение — попарно внутри деления:**
- Вертикальный сплит: оба потомка получают равную ширину (50%/50% от ширины родителя)
- Горизонтальный сплит: оба потомка получают равную высоту (50%/50% от высоты родителя)

Глубина дерева не ограничена, но на практике не более 3–4 уровней. Пример: ячейка B2 в 9 суб-ячеек (3 вертикальных × 3 горизонтальных уровня):

```
Верхний ряд:    B2(1,1) B2(1,2) B2(1,3)   — 50% высоты, 33% ширины каждая
Средний ряд:    B2(2,1) B2(2,2) B2(2,3)   — 25% высоты, 33% ширины каждая
Нижний ряд:     B2(3,1) B2(3,2) B2(3,3)   — 25% высоты, 33% ширины каждая
```

### Адресация суб-ячеек
Суб-ячейки адресуются как `{базовая_ячейка}({строка},{столбец})` внутри суб-сетки, нумерация с 1, отсчёт сверху-слева.

Примеры: `B2(1,1)`, `B2(2,3)`, `A1(1,2)`.

### Один товар на листовую ячейку
Базовая ячейка без делений — это листовая ячейка. Товар назначается только на листовую ячейку. Одна листовая ячейка — один товар.

### Объединение (merge) — только для siblings BSP
Можно объединить только двух потомков одного родителя (siblings в BSP-дереве). Родитель становится листовой ячейкой. Размеры пересчитываются.

### Добавление строк и столбцов — только append
Добавление строки или столбца возможно только с края (не вставка в середину). Строка добавляется снизу, столбец — справа. Адреса существующих ячеек не меняются.

### Объединение базовых ячеек
Две соседние (по горизонтали или вертикали) базовые ячейки можно объединить при соблюдении условий:
- **Ни в одной не должно быть BSP-сплитов** (заблокировано, если есть суб-ячейки)
- Если у одной или обеих есть назначенный товар — показывается предупреждение, товар удаляется при подтверждении
- Результат: одна ячейка с размерами = сумма по оси объединения

## Модель данных

### Shelf

| Поле | Тип | Описание |
|------|-----|---------|
| `id` | uuid | |
| `name` | string | Название стеллажа |
| `rows_count` | integer | Количество строк (N) |
| `cols_count` | integer | Количество столбцов (M) |

### Cell

Единая таблица для базовых ячеек и всех узлов BSP-дерева.

| Поле | Тип | Обяз. | Описание |
|------|-----|:---:|---------|
| `id` | uuid | ✓ | |
| `shelf_id` | FK → Shelf | ✓ | |
| `parent_id` | FK → Cell | — | null = базовая ячейка (корень BSP) |
| `split_direction` | enum `H`/`V` | — | null пока не сплитована |
| `is_first_child` | boolean | — | true = левая/верхняя; null на корневых |
| `row_index` | integer | ✓ базовые | Строка в базовой сетке (1-based) |
| `col_index` | integer | ✓ базовые | Столбец в базовой сетке (1-based) |
| `width_mm` | integer | ✓ базовые | Физическая ширина ячейки, вводится вручную |
| `height_mm` | integer | ✓ базовые | Физическая высота ячейки, вводится вручную |
| `computed_width_mm` | integer | ✓ все | Кэш: ширина с учётом BSP-делений |
| `computed_height_mm` | integer | ✓ все | Кэш: высота с учётом BSP-делений |
| `product_id` | FK → Product | — | Только на листовых ячейках |
| `capacity_override` | integer | — | Ручное переопределение вместимости (S03) |

**Инварианты:**
- `parent_id IS NULL` ↔ базовая ячейка (имеет `row_index`, `col_index`, `width_mm`, `height_mm`)
- `parent_id IS NOT NULL` ↔ суб-ячейка (наследует `row_index`/`col_index` от корня, не имеет собственных `width_mm`/`height_mm`)
- Листовая ячейка = нет потомков в таблице Cell (может иметь `product_id`)
- Нелистовая ячейка не имеет `product_id`

**Расчёт `computed_width_mm` и `computed_height_mm`:**
Пересчитываются при каждом сплите или мерже. Для листа:
- `computed_width_mm = root.width_mm × (0.5 ^ кол-во_V-предков_от_корня_до_листа)`
  - Уточнение: "кол-во V-предков" = количество V-сплитов на пути от корня к листу
- На практике: после каждого сплита/мержа пересчитываются `computed_*` всех затронутых листьев обходом дерева. Значение хранится как `integer`; floor-округление при делении пополам допустимо.

## Что намеренно за бортом

| Что | Почему |
|-----|--------|
| Несколько стеллажей | Преждевременно для v1 |
| QR-коды на ячейках | Нет места для наклеек; сотруднику проще найти ячейку глазами |
| Undo/redo | Только явный merge как операция откатки |
| Вставка строки/столбца в середину | Ломает адресацию существующих ячеек |
| Объединение базовых ячеек с BSP-сплитами внутри | Слишком сложно, нет практического сценария |
| Глубина товара (длина) в расчёте вместимости | Не ограничивает вместимость; важна как свойство товара для фильтрации (S01) |
| Визуальные пропорции строк/столбцов | Визуально сетка равномерная; физические мм — в данных, не в UI |

## Зафиксировано для S03 (расчёт вместимости)

- **Размеры для расчёта**: `computed_width_mm` и `computed_height_mm` листовой ячейки
- **Глубина товара (length_mm)** не используется в расчёте вместимости ячейки
- Детали формулы, поворота и ручного переопределения — в S03
