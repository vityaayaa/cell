# S08 — История

## Принятые решения

### Что попадает в историю

В историю попадают все сессии: `completed` и `abandoned`. Отображаются вместе в хронологическом порядке с разными значками статуса.

Сессии в фазе `sweeping`, `ordering`, `fulfilling` — не в истории, они активны.

---

### Неизменяемость и удаление

| Действие | Кто | На что |
|----------|-----|--------|
| Удалить | Admin | Только `abandoned` сессии |
| Удалить | Никто | `completed` — запрещено |
| Редактировать | Никто | Любую запись в истории |

`completed` сессии защищены от удаления — иначе статистика по товарам становится недостоверной.

---

### Уровень аналитики

**Уровень 2: список + агрегаты по товарам. Графиков нет.**

Список сессий даёт хронологию. Агрегаты по товарам — статистику. При одной заявке в месяц графики избыточны.

---

### Период агрегатов

Все агрегаты считаются за всё время — по всем `completed` сессиям. Фильтра по периоду нет.

---

### Снимок конфигурации стеллажа

Не сохраняется. Контекст "как выглядел стеллаж тогда" не нужен для истории заявок.

---

### Снимок имени товара

`OrderLine` хранит `product_name` — строку, скопированную из каталога в момент создания заявки.

Причина: если admin удалит товар из каталога, история не ломается — имя уже записано в строке заявки.

---

### Хранение данных

Хранить вечно. При одной сессии в месяц — 12 записей в год. Проблем с объёмом нет.

---

### Экспорт

Excel-файл: сводная таблица по товарам за всё время. Один товар — одна строка.

Экспорт отдельной сессии (детальный список) — out of scope для v1.

---

## Экран истории

### Список сессий

Каждая строка списка показывает:

| Поле | Откуда |
|------|--------|
| Дата начала / завершения | `Session.started_at` / `finished_at` |
| Кто | `UserProfile.name` через `Session.user_id` |
| Статус | `Session.status` — значок "завершена" / "брошена" |
| Кол-во позиций | Кол-во `OrderLine` в `Order` этой сессии (null для abandoned без заявки) |
| Длительность | `finished_at − started_at` (только для completed) |

Список отсортирован: новые сверху.

---

### Детали сессии

Тап на сессию → детальный экран.

**Для `completed`:** таблица позиций с plan vs fact.

| Товар | Заказали (пачки) | Взяли (пачки) | Статус |
|-------|:---:|:---:|:---:|
| Брусок 50×50×3000 | 6 | 6 | ✓ |
| Доска 20×100×6000 | 4 | 3 | частично |
| Труба ПВХ ⌀110×3000 | 2 | — | нет на складе |

Если `actual_packs = quantity_packs` → "взяли всё".  
Если `actual_packs < quantity_packs` → показывается отклонение.  
Если `status = unavailable` → "нет на складе".

**Для `abandoned`:** показывается то, что успели: остатки из обхода и, если есть, незавершённая заявка.

---

### Агрегаты по товарам

Отдельная секция (или вкладка) на экране истории.

По каждому товару, который хотя бы раз встречался в `completed` сессиях:

| Поле | Как считается |
|------|--------------|
| Название | `OrderLine.product_name` |
| Раз в заявке | Кол-во completed сессий, где товар есть в OrderLine |
| Среднее пачек | Среднее `OrderLine.quantity_packs` по всем вхождениям |
| Среднее взяли | Среднее `ChecklistEntry.actual_packs` где `status = done` |
| Раз не было | Кол-во `ChecklistEntry` где `status = unavailable` |

Если "среднее взяли" стабильно ниже "среднего заказали" — склад регулярно недодаёт.

---

## Аналитические запросы, которые история поддерживает

1. **Хронология сессий** — список всех сессий, новые сверху
2. **Детали сессии** — OrderLines + ChecklistEntries для одной сессии
3. **Агрегат по товару** — stats по всем completed сессиям для одного product_name
4. **Сводная таблица** — все агрегаты по всем товарам (для экспорта в Excel)

## Модель данных

### Дополнение к OrderLine (S05)

Добавляется одно поле:

| Поле | Тип | Описание |
|------|-----|---------|
| `product_name` | string | Имя товара на момент создания заявки; копируется из `Product.name` + параметры |

**Полная модель OrderLine:**

| Поле | Тип | Описание |
|------|-----|---------|
| `id` | uuid | |
| `order_id` | FK → Order | |
| `product_id` | FK → Product | null если товар удалён из каталога |
| `product_name` | string | Снимок имени: "Брусок 50×50×3000" |
| `quantity_packs` | integer | Финальное количество пачек |
| `quantity_units` | integer | Штук (= quantity_packs × pack_size) |
| `deficit_units` | integer | Расчётный дефицит (null для ручных строк) |
| `is_manual` | boolean | Добавлено вручную из каталога |
| `is_boundary` | boolean | Активированная пограничная позиция |

Новых таблиц нет — история строится на существующих `Session`, `Order`, `OrderLine`, `ChecklistEntry`.

---

## Экспорт

Формат: Excel (.xlsx). Один файл — сводная таблица по всем товарам за всё время.

Столбцы:
- Название товара
- Раз в заявке
- Среднее пачек (заказано)
- Среднее пачек (взято)
- Раз не было на складе

Экспорт доступен из экрана агрегатов. Генерируется на клиенте из уже загруженных данных.

## Что намеренно за бортом

| Что | Почему |
|-----|--------|
| Снимок конфигурации стеллажа | Не нужен для истории заявок |
| Графики и тренды | Уровень 2 достаточен; 1 сессия/месяц — данных мало |
| Фильтр по периоду | Всегда за всё время; усложнение без практической нужды |
| Удаление completed сессий | Защита целостности статистики |
| Редактирование истории | История неизменяема |
| Экспорт отдельной сессии | Достаточно сводной таблицы для v1 |
| Predictive analytics / forecasting | Выходит за рамки v1 |
