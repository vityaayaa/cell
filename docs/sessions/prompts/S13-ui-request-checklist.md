## Что принести в разговор

- **Swipe to complete**: стандарт для мобильных чеклистов (iOS Reminders, Todoist). Но на складе в перчатках — работает ли? Альтернатива: большая кнопка "Взял" справа.
- **Визуальное разделение "взял / не взял / нет на складе"**: три состояния должны быть мгновенно различимы без чтения текста. Цвет + иконка — не только цвет (colour blindness).
- **Sticky header с прогрессом**: "7 из 12 позиций взято" всегда видно сверху — мотивирует и информирует.
- **"Поделиться" чеклистом**: отправить ссылку/PDF на другой телефон для второго сотрудника. Реальный кейс?
- **Undo для случайных отметок**: "Случайно нажал 'Нет на складе'" — кнопка отмены или только если идти обратно в элемент?
- **Группировка в чеклисте**: по зоне склада, по типу товара, по статусу. Длина как первичная сортировка — для погрузки на технику, но не для навигации по складу.

## Ты — критик: оспорь возможные решения

1. **Заявка и чеклист — это один экран или два?**
   Если пользователь находится в заявке и нажимает "Финализировать" → переходит к чеклисту: это нормально. Но если это один экран с вкладками "Заявка / Чеклист" — это путает границы состояний (черновик vs. финализировано). Стоит явно разделить.

2. **Редактирование количества в черновике inline — удобно или опасно?**
   Tap на цифру → меняешь число. Легко случайно изменить при прокрутке. Может, лучше: кнопка "Редактировать" → режим редактирования → "Сохранить"? Медленнее, но надёжнее.

3. **Кнопка финализации — насколько она "заметная"?**
   Если она слишком заметная, нажимают случайно. Если слишком спрятанная, не находят. Confirmation dialog ("Вы уверены?") — кажется защитой, но пользователи кликают "OK" не читая. Как защититься реально?

## Уже закрыто — не переобсуждать

### Из S05 (заявка) и S06 (чеклист)
- **Заявка и чеклист — разные фазы, не вкладки**: `ordering` → `fulfilling` — переход односторонний (назад нельзя). Из `ordering` можно вернуться в `sweeping`, но не обратно из `fulfilling`.
- **Статусы строки чеклиста**: `pending` | `done` | `unavailable` — зафиксированы в S05.
- **`actual_packs`**: при `done` фиксирует сколько реально взяли. Если меньше `quantity_packs` — история показывает "заказали N, взяли M" (S06).
- **Сортировка**: материал (выбранный первым) → длина убывает → название А→Я. Одинаковая в заявке и чеклисте (S05).
- **Строки `done`/`unavailable` уходят в конец** автоматически, внутри — та же сортировка (S06).
- **Прогресс "Взято N из M"** — sticky header в чеклисте (S06).
- **Завершение сессии**: автоматически при всех строках в `done` или `unavailable` (S05/S06).
- **Бумажный PDF-экспорт** из чеклиста — через печать браузера, без отдельной библиотеки (S06/S10).
- **Нужно показывать количество**: `quantity_packs` виден в каждой строке чеклиста (S06 — "что взял всё" vs "взял меньше" требует числа).

### Из S12 (UI ввода остатков)
- **Toast с прогресс-баром** — глобальное правило (S00), не нужно решать в S13.
- **Переход к заявке**: кнопка "→ К заявке" в строке прогресса обхода. Открывает черновик заявки (фаза `ordering`).
