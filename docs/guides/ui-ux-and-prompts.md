# Гайд: UI/UX и работа с промптами в проекте CELL

Практическая памятка. Две части: **(A)** как делать интерфейс действительно удобным именно для этого приложения, **(B)** как вести UI-сессии и работать с промптами, чтобы не терять качество.

Все цифры — из устоявшихся стандартов (Apple HIG, Google Material, WCAG). Это не «на память», а отраслевые ориентиры.

---

# A. UI/UX для CELL

## A0. Главное правило: проектируй под худший контекст

У приложения два пользователя в очень разных условиях:

| Пользователь | Где и как | Следствие для дизайна |
|--------------|-----------|----------------------|
| **Сотрудник** | Стоит у стеллажа. Телефон в одной руке. Перчатки, яркий свет, шум, спешка | Крупные кнопки, большой контраст, минимум точности, работа большим пальцем одной руки |
| **Админ** | За столом или дома, изредка | Может позволить плотные таблицы и формы |

**Проектируй сотруднику под стеллажом.** Если удобно ему — удобно всем. Админские экраны (каталог, настройка стеллажа, аккаунты, история действий) можно делать плотнее — они для спокойного контекста.

## A1. Размеры нажатий (это про перчатки)

- Минимум по стандартам: **44×44 px** (Apple HIG), **48×48 dp** (Material), **44×44** (WCAG 2.5.5 AAA). Нижняя граница для AA — 24×24 (WCAG 2.5.8), но это слишком мало для нас.
- **Для CELL: основные действия — 56–64 px**, потому что перчатки и спешка. «Взял», «Сохранить», тап по ячейке — крупные.
- **Промежуток между кнопками ≥ 8 px**, чтобы не попасть в соседнюю.
- Опасные действия (финализация, удаление) — **не рядом** с частыми, чтобы не задеть.

## A2. Зона большого пальца

- Частые действия — **в нижней трети экрана** (туда дотягивается палец одной руки).
- Верхние углы — для редкого (назад, меню), не для основного.
- Нижняя панель/кнопки лучше, чем верхние, для рабочих экранов (обход, чеклист).

## A3. Цвет — никогда в одиночку

~8% мужчин не различают красный/зелёный. Поэтому **статус = иконка + форма + подпись + цвет**, а не только цвет.

Цветовая схема остатков (из S12): зелёный «достаточно» → жёлтый «мало» → красный «критично» → серый «нет данных». Но рядом — иконка и/или число, чтобы читалось и без цвета и на солнце.

## A4. Мгновенная реакция и оптимистичный UI

- На любой тап — видимый отклик **меньше чем за 0.1 сек** (нажатие, подсветка).
- **Оптимистичный UI** (заложено в S09/S10): запись показывается сразу, синхронизация — в фоне. Пользователь не ждёт сервер.
- Это совпадает с «витриной» из S10: экран читает из локальной копии — поэтому всё мгновенно.

## A5. Офлайн и синхронизация — видно, но не мешает

- Состояния из S09: `онлайн / офлайн / синхронизация`.
- Показывать **ненавязчивой полоской/значком**, не модальным окном. Сотрудник продолжает работать офлайн — приложение не должно его блокировать.
- «Не синхронизировано N действий» — спокойная информация, не тревога.

## A6. Пустые экраны и первый запуск

- Пустой стеллаж/каталог при первом запуске **не оставлять голым** — это отталкивает (см. S14, онбординг).
- Пустое состояние = подсказка к следующему шагу: «Добавьте первый товар», «Настройте стеллаж».

## A7. Ошибки — человеческим языком и с действием

- Плохо: «Error 500», «Не удалось».
- Хорошо: «Не сохранилось — нет связи. Сохраним автоматически, когда появится интернет.»
- Всегда: что случилось + что будет дальше / что сделать.

## A8. Ввод чисел

- Числовые поля — **числовая клавиатура** (`inputmode="numeric"`).
- Размер шрифта в полях **≥ 16 px** — иначе iOS Safari автоматически зумит страницу при фокусе (раздражает).
- «Слепой пересчёт» (S04): прошлый остаток не показываем — поле пустое. Это сознательно, не баг.
- Валидация (S04): остаток > вместимости — блок с понятным сообщением, не молчаливый сброс.
- Для навалом — рассмотреть кнопки «0 / ¼ / ½ / ¾ / Полная» вместо слайдера (слайдер плох в перчатках — это разберём в S12).

## A9. Защита необратимых действий

Подтверждение «Вы уверены? [OK]» **не работает** — жмут не читая. Надёжнее:
- двухшаговое действие (нажать → появляется отдельная подтверждающая кнопка в другом месте),
- или удержание (hold-to-confirm),
- или ввод/свайп для подтверждения.

Касается: финализация заявки (S05), удаление товара/ячейки/пользователя (S07).

## A10. Анимации (Motion) — со смыслом

- Цель — **объяснять изменение состояния**, а не украшать. Пример: отмеченная позиция уезжает вниз списка (S06) — анимация показывает, что произошло.
- Быстро: **150–250 мс**. Длинные анимации раздражают при частой работе.
- Уважать `prefers-reduced-motion` — у кого укачивает, отключить.

## A11. Контраст и читаемость на солнце

- Текст: контраст **≥ 4.5:1** (WCAG AA), крупный текст ≥ 3:1.
- Базовый шрифт крупный, высокий контраст — экран читают при ярком свете.
- Минимум декора, максимум ясности — это стандарт для «полевых»/складских приложений.

## A12. Мобильные мелочи, которые часто забывают

- **Safe-area insets** — отступы под «чёлку» и нижнюю полоску iPhone (`env(safe-area-inset-*)`).
- **`100dvh` вместо `100vh`** — иначе нижняя панель браузера перекрывает контент.
- Запретить случайный зум/«резинку» там, где мешает.
- Проверять на **реальном телефоне**, не только в десктопном браузере.

## A13. Единый язык по всему приложению

- Одна сортировка везде (S05): материал → длина убыв. → имя А-Я. В заявке и чеклисте — одинаково.
- Ячейка выглядит одинаково в конструкторе (S11) и при обходе (S12) — один визуальный язык, разные режимы.
- Одни и те же иконки статусов везде.

## A14. Чек-лист перед тем, как считать экран готовым

- [ ] Основные кнопки ≥ 56 px, промежутки ≥ 8 px
- [ ] Частые действия — в зоне большого пальца
- [ ] Статусы различимы без цвета (иконка + подпись)
- [ ] Каждый тап даёт мгновенный отклик
- [ ] Работает офлайн, состояние связи видно ненавязчиво
- [ ] Пустое состояние подсказывает следующий шаг
- [ ] Ошибки — человеческим языком + что дальше
- [ ] Числовые поля: numeric-клавиатура, шрифт ≥ 16px
- [ ] Необратимое действие защищено не одним «OK»
- [ ] Контраст ≥ 4.5:1, читается на солнце
- [ ] Учтены safe-area и `100dvh`
- [ ] Проверено на реальном телефоне

# B. Как вести UI-сессии и работать с промптами

## B1. Рабочий цикл, который уже доказал себя в этом проекте

1. **Прочитать ВСЕ файлы из «Прочитай перед началом»** — без исключений (мета-правило CLAUDE.md). Неполное чтение = неверные ответы.
2. **Исследование ДО разговора** — Context7 (версии/синтаксис), Exa (сравнения, community, «почему плохо»). Не на память.
3. **Принести больше, чем в промпте** — после ресёрча спросить себя: «что здесь важного и неочевидного, чего нет в промпте?»
4. **Критик** — оспорить собственные и чужие предположения, назвать trade-off каждого выбора.
5. **Вопросы пачкой**, но по одному крупному решению за ответ — итерация, не монолит.
6. **Спека с обоснованиями** + раздел «что намеренно отклонено».
7. **Вперёд:** в промпт следующей сессии добавить «Уже закрыто», чтобы не переобсуждать.

## B2. Особенность UI-сессий (S11–S14)

- Это **спек-сессии, не кодинг.** Решаем поведение и структуру экранов, а не пишем компоненты.
- **Нащупать структуру и потоки важнее, чем визуал.** Цвета и отступы потом меняются за минуты; логика экранов и переходы — дорого переделывать.
- Обсуждать на **реальных примерах и числах**, не абстракциями (пользователь — не кодер: псевдокод и формулы не помогают, только слова и примеры).

## B3. Держать промпты честными

- Если поздний промпт противоречит ранней спеке — **назвать противоречие и решить**, не идти молча. Пример: QR-коды в S11/S12 против запрета QR в S02 (уже помечено в S11).
- Перед сессией проверять, что список «Прочитай» полный и пути верные.
- Вопросы, уже закрытые в S01–S10, помечать в «Уже закрыто», чтобы не жечь время.

## B4. Технические факты — только проверенные

- Версии, возможности и синтаксис библиотек — через **Context7** (ключ настроен глобально в `~/.claude/.mcp.json`). Не из памяти.
- Сравнения подходов и «когда не стоит» — через **Exa**.
- Готовые реализации — через **GitHub MCP**.

## B5. Шаблон обсуждения одного экрана (можно переиспользовать)

```
Экран: <название>
Кто пользуется: <роль> в <контексте>
Задача пользователя здесь: <одна фраза>
Главное действие (одно): <...>
Второстепенные действия: <...>
Состояния: пустое / загрузка / офлайн / ошибка / успех
Что уже решено в спеках: <ссылки на S0X>
Открытые вопросы (trade-off каждого): <...>
```

## B6. Когда все 14 спек готовы — переход к коду

Порядок реализации (по архитектуре S10):
1. **Доменное ядро** (`domain/`) — формула вместимости, BSP, расчёт заявки — **с тестами в первую очередь** (тут тихая ошибка = неверная заявка).
2. **Слой данных** (`data/`) — схема Dexie, клиент Supabase, синхронизация, офлайн-очередь.
3. **Экраны** (`features/`) — по одному, от настройки к ежедневному обходу.
4. План — через skill `writing-plans` (указано в конце S14), затем реализация по модулям.

## B7. Короткие правила-напоминания

- Одна спека — одна тема. После блока — «Переходим?».
- Нет рекомендации без её цены (trade-off).
- Не уверен — «не знаю, проверю», и проверяешь.
- Ошибся раньше — исправить явно, не защищать по инерции.
- Сложную логику (вместимость) — покрывать тестами, не «на глаз».
